Quick Start

Tensō makes it very easy to create API facades. Routes should be loaded as a module, with each HTTP method as an export; this enforces a separation of concerns by convention. However, sometimes you want routes to be declared in your core application file, for access to lexical scope variables, such as the configuration Object.

Routes can have Booleans, Strings, Numbers, Arrays, & Functions (referred to as 'handlers'). When a route doesn't have a handler, the value is automagically serialized and sent to the Client. Routes that have handlers need to only focus on their task(s), and eventually "respond" with a "result"; status code and custom HTTP headers are optional.

Route handlers have the context of the Tensō server, i.e. this will allow you to send a response with this.respond(req, res, body[, status, headers]), and this.error(req, res, status[, Error]). You can also use res to res.respond(body[, status, headers]), res.redirect(url), or res.error(status[, Error]).

Responses

Responses will have a standard shape, and will be utf-8 by default. The result, and hypermedia will be in data. Hypermedia (pagination, links, etc.) will be in data.link, & pagination will also be present via the Link HTTP header. The result will be in data.result.

Helpers

Tensō decorates req with "helpers" such as req.allow, req.ip, & req.parsed. PATCH, PUT, & POST payloads are available as req.body. Sessions are available as req.session when using certain types of authentication.

Horizontal scaling

An easy way to scale when using bearer tokens, local auth, or a social auth, is to use Redis for your token/session storage, and the cluster module or multiple servers.
